1. Die Software-Architektur (Die 4 Kern-Schichten)Das Programm ist in vier logische Schichten unterteilt, die modular zusammenarbeiten:A. Der Data Layer (Das physikalische Modell)Code-Anker: getVelocityAt(x, y, z) und getColorForMagnitude(mag)Funktion: Dies ist das mathematische Herzstück. Es simuliert die Strömungsmechanik (CFD). Das Programm fragt für jede beliebige 3D-Koordinate $(x, y, z)$ den dort anliegenden Geschwindigkeitsvektor ab.Forschungs-Vorteil: Die Grafik-Engine weiß nichts von Physik. Wenn du das Modell ändern willst (z. B. von einem Wirbel zu einer laminaren Rohrströmung), musst du nur diese eine mathematische Funktion austauschen.B. Der Engine Layer (Die 3D-Infrastruktur)Code-Anker: init(), THREE.Scene, THREE.PerspectiveCamera, THREE.WebGLRendererFunktion: Dieser Teil baut das "Labor" auf. Er initialisiert die Grafikkarte über WebGL, stellt die Kamera auf, platziert die Beleuchtung und zeichnet die graue Bounding-Box (den Forschungsraum), damit der Wissenschaftler eine räumliche Orientierung hat.C. Der Pipeline Layer (Die Geometrie-Generatoren)Code-Anker: generateGlyphs(), generateStreamlinesAndRibbons()Funktion: Hier findet die Übersetzung von reinen Zahlendaten in sichtbare 3D-Objekte statt.Die Glyphen-Pipeline läuft in einer dreidimensionalen Schleife durch den Raum und platziert THREE.ArrowHelper-Objekte.Die Streamline-Pipeline nutzt das Euler-Verfahren (numerische Integration): Sie setzt ein virtuelles Partikel an einen Startpunkt, fragt den Data Layer nach der Geschwindigkeit, bewegt das Partikel ein Stück vorwärts (addScaledVector) und wiederholt das 70 Mal. Aus diesen Punkten wird eine glatte Kurve (CatmullRomCurve3) generiert.D. Der Control Layer (Das User Interface)Code-Anker: setupLayerToggles(), HTML/CSS Control-PanelFunktion: Dieser Layer verbindet die HTML-Checkboxen des Dashboards mit der 3D-Szene. Er fängt die Klicks des Nutzers ab und steuert die Sichtbarkeit der Daten.2. Der Programmlauflauf (Lifecycle)Wenn die Datei im Browser geöffnet wird, durchläuft das Programm exakt diese Sequenz:[1. Start] -> [2. init() Engine Setup] -> [3. Geometrie-Berechnung] -> [4. Event-Listener] -> [5. Endlos-Render-Schleife]
Infrastruktur aufbauen (init): Die Szene wird erstellt. Wichtig: Hier werden drei leere Container (THREE.Group()) für Glyphen, Streamlines und Ribbons in die Szene eingehängt.Daten generieren: Die Generatoren berechnen die Pfeile, Linien und Bänder und stopfen sie in die jeweiligen Container-Gruppen.Interaktion scharfschalten: Die Event-Listener warten auf Checkbox-Änderungen.Die Render-Schleife (animate): Das Programm startet eine Endlosschleife über requestAnimationFrame. Etwa 60 Mal pro Sekunde wird geprüft, ob der Nutzer die Kamera mit der Maus bewegt hat (controls.update()), und das Bild wird neu gezeichnet (renderer.render()).3. Clevere Speicher- und Performance-Tricks im CodeDa wissenschaftliche Datensätze riesig sein können, nutzt das Programm zwei wichtige Optimierungen:Gruppen-Sichtbarkeit statt Objekt-Löschung: Wenn du die Glyphen ausschaltest, löscht das Programm die Tausenden Pfeile nicht aus dem Arbeitsspeicher. Es setzt lediglich glyphsGroup.visible = false. Dadurch weiß die Grafikkarte sofort: "Diesen Block beim Zeichnen überspringen". Das Umschalten passiert ohne jegliche Verzögerung (0 Millisekunden Ladezeit).Vertex-Coloring (Farbattribute): Bei den Streamlines werden die Farben (Blau bis Gelb) als Attribut direkt in die Punkte der Geometrie geschrieben (streamGeometry.setAttribute('color', ...)). Das bedeutet, die Grafikkarte berechnet den Farbverlauf extrem effizient direkt im Shader, anstatt dass für jedes Liniensegment ein eigenes Grafikmaterial erstellt werden muss.
